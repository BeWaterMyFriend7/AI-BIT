# 2. 背景与需求描述

## 2.1 背景问题

用户在同一台 Windows 机器上同时使用 Codex 与 OpenCode，二者默认 Skill 目录不同：

> Codex: `C:\Users\BeWater\.codex\skills`
>
> OpenCode：`C:\Users\BeWater\.config\opencode\skills`

目前通过 PowerShell 脚本按 Skill 建立 NTFS Junction，可以避免重复安装和重复维护。但随着 Agent 和 Skill 数量增加，纯脚本方式存在以下问题：

- 无法直观看到每个 Agent 已安装了哪些 Skill，以及同名 Skill 是否为同一份内容。

- 新增 Agent 或调整 Skill 目录时，需要手工修改脚本。

- 启用、停用共享缺少图形化交互和状态反馈。

- 同名真实目录、断链、源目录迁移等异常场景不容易发现。

- 误删真实目录或覆盖已有副本可能造成不可逆损失。

## 2.2 总体需求

> **总体需求**：提供一个仅本机访问的 Web 页面，以 Skill 为主视角统一管理不同 Agent 之间的 Skill 可用性。用户可配置各 Agent 的 Skill 根目录，系统自动识别已安装 Skill，并通过点击 Agent 图标启用或取消指定 Skill 的共享。

## 2.3 用户原始需求映射

| **编号** | **原始需求**                                                      | **产品化解释**                                                                 |
|----------|-------------------------------------------------------------------|--------------------------------------------------------------------------------|
| R-01     | 本地访问的 Web 页面进行交互和选择，管理不同 Agent 间的 Skill 共享 | 启动本地 HTTP 服务，仅监听 127.0.0.1，浏览器完成配置、扫描、启停和查看。       |
| R-02     | 支持对不同 Agent Skill 目录的配置                                 | 支持新增、编辑、停用 Agent，配置名称、图标、Skill 根目录、忽略规则和链接方式。 |
| R-03     | 支持识别当前 Agent 已安装的 Skill                                 | 扫描各根目录的一级子目录及 SKILL.md，识别真实目录、Junction、断链和冲突副本。  |
| R-04     | 按 Skill 维度查看列表，右侧显示不同 Agent 图标，点击启用或不共享  | 主页面采用 Skill × Agent 状态矩阵；图标既显示状态，也是操作入口。              |

# 3. 产品目标与范围

## 3.1 产品目标

- 实现“一份 Skill，多 Agent 使用”，避免重复复制和版本漂移。

- 提供可视化 Skill × Agent 矩阵，快速识别安装状态和共享状态。

- 将 PowerShell Junction 脚本封装为安全、可回滚的本地服务操作。

- 新增 Agent 时只需配置目录，无需修改核心代码。

- 任何破坏性操作均需明确确认，默认备份而非删除真实目录。

## 3.2 首期范围（MVP）

| **范围** | **包含内容**                                        |
|----------|-----------------------------------------------------|
| 平台     | Windows 10/11，NTFS 文件系统。                      |
| Agent    | 内置 Codex、OpenCode 预设；支持自定义 Agent。       |
| 共享方式 | 逐个 Skill 创建或移除 NTFS Junction。               |
| 入口     | 本地 Web 页面；默认 `http://127.0.0.1:17890`。        |
| 数据存储 | 本地 SQLite 或 JSON 配置；不依赖云服务。            |
| 安全     | 仅监听 loopback；路径白名单；操作日志；备份与回滚。 |

## 3.3 暂不纳入首期

- 跨机器、局域网或云端同步。

- 在线 Skill 市场、下载、升级和依赖管理。

- 自动修改 Agent 自身配置文件。

- Git 仓库同步、团队权限和多用户协同。

- 对未知 Skill 格式进行内容级修复。

# 4. 用户场景

| **场景**     | **操作路径**                                                 | **预期结果**                                                |
|--------------|--------------------------------------------------------------|-------------------------------------------------------------|
| 首次使用     | 启动程序 → 打开 Web 页 → 使用 Codex/OpenCode 预设路径 → 扫描 | 显示两个 Agent 和所有已安装 Skill。                         |
| 启用共享     | 在某 Skill 行点击 OpenCode 图标                              | 若 OpenCode 无该 Skill，则创建 Junction；状态变为“已共享”。 |
| 取消共享     | 再次点击已共享图标并确认                                     | 仅删除 Junction，不删除源 Skill；状态变为“不可用”。         |
| 发现独立副本 | 扫描后某 Agent 图标显示蓝色“本地副本”                        | 用户可保留独立副本，或备份后转换为共享。                    |
| 同名内容冲突 | 两个 Agent 均有同名真实目录且指纹不同                        | 显示冲突，禁止直接覆盖，要求选择源或比较后处理。            |
| 增加 Agent   | 设置页新增 Claude Code，填写 Skill 根目录和图标              | 重新扫描后矩阵新增一列。                                    |
| 源目录失效   | 源 Skill 被移动或删除                                        | 关联 Agent 显示“断链”，禁止继续切换，并提供重新指定源。     |

主页面以 Skill 为行、Agent 为列。左侧展示 Skill 基本信息，右侧每个 Agent 以图标或徽标展示当前状态；点击图标触发启用、取消或冲突处理。

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ SkillBridge  [搜索 Skill...] [全部状态▼] [重新扫描] [Agent 设置]            │
├──────────────────────────────────────────────────────────────────────────────┤
│ Skill                         Codex         OpenCode      Claude Code          │
├──────────────────────────────────────────────────────────────────────────────┤
│ semantic-merge-review         [源 🔒]       [共享 ✓]       [未启用 ○]          │
│ semantic-merge-executor       [源 🔒]       [共享 ✓]       [未启用 ○]          │
│ superpowers                   [本地 ●]      [冲突 !]       [共享 ✓]            │
│ experimental-skill            [未启用 ○]    [断链 ×]       [未启用 ○]          │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 列表字段

| **字段**   | **说明**                                                      |
|------------|---------------------------------------------------------------|
| Skill 名称 | 优先读取 `SKILL.md` frontmatter 的 name；读取失败时使用目录名。 |
| 描述       | 读取 description；未提供时显示“无描述”。                      |
| 源标识     | 显示当前真实源所在 Agent 或“中央源”。                         |
| Agent 图标 | 显示该 Skill 在指定 Agent 中的状态，并作为操作按钮。          |
| 更新时间   | 源目录最近修改时间或最近扫描时间。                            |
| 异常标记   | 冲突、断链、无 `SKILL.md`、目录不可访问等。                     |

| **状态**     | **视觉建议**            | **点击行为**                              |
|--------------|-------------------------|-------------------------------------------|
| SOURCE_LOCAL | 绿色实心 + 锁或“源”角标 | 打开详情；不能直接关闭，需先迁移源。      |
| SHARED_LINK  | 绿色勾选                | 弹出取消共享确认；确认后只移除 Junction。 |

| BROKEN_LINK  | 红色叉号                | 重新指定源、修复联接或删除断链。          |
| INVALID      | 灰红色禁用              | 查看 `SKILL.md`、权限或目录结构错误。       |

# 7. Skill 识别与状态模型

## 7.1 Skill 唯一标识规则

1.  优先使用 `SKILL.md` frontmatter 中的 name。

2.  `SKILL.md` 不存在或解析失败时，使用目录名作为 provisional_name，并标记 INVALID。

3.  同一个 Agent 根目录下不允许出现两个相同 name。

4.  跨 Agent 的同名目录进入同一 Skill 聚合项，但必须比较物理路径、联接目标和内容指纹。

## 7.2 内容指纹

为判断两个真实目录是否为同一内容，计算目录指纹。MVP 建议对以下内容进行稳定排序后计算 SHA-256：相对路径、文件大小、最后修改时间；在冲突确认时再执行全文件哈希。忽略 `.git`、缓存、临时文件和配置中的忽略规则。

## 7.3 状态判定

| **条件**                                   | **状态**     |
|--------------------------------------------|--------------|
| 目录为真实目录，且被选为源                 | SOURCE_LOCAL |
| 目录为 Junction，最终目标等于选定源        | SHARED_LINK  |
| 目标 Agent 下无该目录                      | MISSING      |
| 目录为真实目录，但不是选定源，内容与源一致 | LOCAL_COPY   |
| 目录为真实目录，且内容与源不同             | CONFLICT     |
| 目录为 Junction，但目标不存在或无法访问    | BROKEN_LINK  |
| 缺少 `SKILL.md`、name 不合法或权限不足       | INVALID      |

## 7.4 源选择规则

MVP 延续现有脚本思路：以一个真实 Agent Skill 目录作为源，例如 Codex 中的 Skill 目录；其他 Agent 通过 Junction 共享。

- 若仅有一个真实目录，则自动设为源。

- 若存在一个真实目录和多个指向它的 Junction，则该真实目录为源。

- 若存在多个内容一致的真实目录，提示用户选择源，其他目录可备份后转为 Junction。

- 若存在多个内容不同的真实目录，标记 CONFLICT，必须人工选择源或保留为不同名称。

- 源所在 Agent 的图标不可直接关闭；需先执行“迁移源”。

# 8. 主要实现思路

## 8.1 总体架构

```text
浏览器（127.0.0.1）
        │ HTTP / JSON
        ▼
本地 Web 服务
├── Agent 配置管理
├── Skill 扫描与聚合
├── 状态判定与冲突检测
├── 共享操作编排
├── 操作日志与回滚
└── Windows Junction 适配器
        │
        ├── C:\Users\BeWater\.codex\skills
        ├── C:\Users\BeWater\.config\opencode\skills
        └── 其他 Agent Skill 目录
```

## 8.2 推荐技术栈

| **层次** | **推荐方案**               | **理由**                                             |
|----------|----------------------------|------------------------------------------------------|
| 后端     | Go 1.22+ 本地服务          | 可编译为单个 exe，启动简单；文件系统与进程控制稳定。 |
| 前端     | React + TypeScript + Vite  | 适合状态矩阵、筛选、弹窗和批量操作。                 |
| 静态资源 | 编译后嵌入 Go 二进制       | 无需单独部署 Web 服务。                              |
| 存储     | SQLite                     | 保存 Agent 配置、Skill 快照、操作日志和回滚记录。    |
| 系统操作 | PowerShell Junction 适配器 | 复用现有已验证脚本逻辑；无需管理员权限。             |
| 通信     | 仅本机 HTTP API            | 前后端清晰解耦，便于后续桌面壳或 CLI 接入。          |

## 8.3 共享启用算法

启用共享时，不复制 Skill 内容，而是在目标 Agent 的 Skill 根目录中创建一个与 Skill 同名的 Junction。流程严格复用此前 PowerShell 脚本的处理原则：

1. 校验源目录存在、包含有效 `SKILL.md`，并位于已配置的合法 Skill 根目录内。

1. 校验目标 Agent 根目录存在；不存在时仅在用户明确确认后创建。

1. 若目标路径不存在，直接创建 Junction。

1. 若目标路径已是指向正确源的 Junction，则返回幂等成功。

1. 若目标路径是其他 Junction 或断链，先提示修复或替换。

1. 若目标路径是真实目录，默认不覆盖；用户选择“备份并转共享”后，移动到备份目录，再创建 Junction。

1. 创建后重新解析 Junction 目标，确认与源目录一致，再更新状态。

```powershell
$source = "C:\Users\BeWater\.codex\skills\semantic-merge-review"
$target = "C:\Users\BeWater\.config\opencode\skills\semantic-merge-review"

# 目标为真实目录时先备份；目标为旧 Junction 时只移除链接本身
if (Test-Path $target) {
    $item = Get-Item -LiteralPath $target -Force

    if ($item.Attributes -band [System.IO.FileAttributes]::ReparsePoint) {
        Remove-Item -LiteralPath $target -Force
    } else {
        $backup = "$target.backup-$(Get-Date -Format yyyyMMdd-HHmmss)"
        Move-Item -LiteralPath $target -Destination $backup
    }
}

New-Item -ItemType Junction -Path $target -Target $source | Out-Null
```

## 8.4 取消共享算法

1. 确认目标路径为 ReparsePoint/Junction。

1. 解析 Junction 最终目标，确认它指向当前 Skill 的选定源。

1. 只删除 Junction 本身，不递归删除源目录。

1. 若目标为真实目录，则不执行删除，提示其为独立副本。

1. 删除后重新扫描，状态应变为 MISSING。

## 8.5 备份规则

为避免覆盖已有 Skill，所有真实目录替换操作必须先移动到目标 Agent 根目录下的专用备份区：

`<agent-skill-root>\skillbridge-backups\<yyyyMMdd-HHmmss>\<skill-name>\`

备份记录写入操作日志，至少包含原路径、备份路径、源路径、目录指纹和操作时间。MVP 不自动清理备份。

## 8.6 目录扫描算法

1. 加载已启用的 Agent 配置。

1. 遍历每个 Skill 根目录的一级子目录，排除 `.system`、备份目录和配置忽略项。

1. 使用文件属性识别真实目录和 ReparsePoint。

1. 解析 Junction 最终目标，检测断链。

1. 读取 `SKILL.md` frontmatter 的 name、description、compatibility 等字段。

1. 建立 AgentSkillInstance 记录并计算快速指纹。

1. 按 name 聚合，推断源目录和各 Agent 状态。

1. 将扫描快照返回前端，并保存最近一次扫描结果。

## 8.7 文件操作事务与补偿

文件系统操作不具备数据库事务能力，因此采用“步骤日志 + 补偿动作”模式。每个操作先生成 Operation 记录，再逐步执行；每一步保存 before/after 状态。失败时按相反顺序执行补偿，例如删除新建 Junction、将备份目录移回原位。

# 11. 安全与非功能需求

## 11.1 安全要求

- 服务默认只监听 `127.0.0.1`，不监听 `0.0.0.0`。

- 前端请求必须携带启动时生成的本地会话令牌，降低其他本机页面发起 CSRF 的风险。

- 所有输入路径必须规范化，并确认位于已配置的 Agent Skill 根目录或备份目录中。

- 不允许对磁盘根目录、用户主目录或 Agent 根目录本身执行删除操作。

- 取消共享只允许删除已验证的 Junction；真实目录必须走备份流程。

- PowerShell 参数通过安全参数传递，不拼接未转义的用户输入。

- 每个修改操作记录审计日志，错误时保留完整诊断信息。

## 11.2 性能与可用性

| **指标**     | **目标**                                                             |
|--------------|----------------------------------------------------------------------|
| 启动时间     | 普通 Windows 机器上小于 3 秒。                                       |
| 页面首次加载 | 小于 2 秒，不含首次全量文件哈希。                                    |
| 扫描规模     | 1000 个 Skill、10 个 Agent 内完成快速扫描不超过 10 秒。              |
| 单项共享操作 | 一般小于 2 秒。                                                      |
| 幂等性       | 重复启用已共享 Skill 不产生重复目录或备份。                          |
| 故障恢复     | 进程中断后可通过重新扫描恢复真实状态；未完成操作标记为 INTERRUPTED。 |

## 11.3 兼容性

首期以 Windows Junction 为准。架构中保留 LinkAdapter 接口，后续 macOS/Linux 使用 symbolic link 实现。不同 Agent 对 SKILL.md 格式的差异只做兼容读取，不自动修改 Skill 内容。

# 12. 异常处理规则

| **异常**                   | **处理方式**                                       |
|----------------------------|----------------------------------------------------|
| Skill 根目录不存在         | 提示创建；未经确认不自动创建。                     |
| 目录无读取权限             | 标记 Agent 不可用，禁止执行共享。                  |
| 目录无写入权限             | 允许扫描，但禁用图标操作。                         |
| 同名真实目录内容不同       | 标记 CONFLICT，要求选源或重命名。                  |
| Junction 目标不存在        | 标记 BROKEN_LINK，可删除断链或重新指定源。         |
| 源目录正在被占用           | 操作失败，不修改目标；展示系统错误。               |
| 创建 Junction 失败         | 恢复已备份目录，操作状态为 ROLLED_BACK 或 FAILED。 |
| 备份目录冲突               | 使用操作 UUID 或毫秒时间戳生成唯一目录。           |
| SKILL.md name 与目录名不同 | 以 name 聚合，并在详情页提示差异。                 |

# 16. 关键设计结论

1. 共享的基本单元是单个 Skill，而不是整个 Agent Skill 根目录。这样不会把 Codex 的 `.system` 等专属内容暴露给其他 Agent。

2. 页面开关背后必须是可验证的目录状态机，而不是简单记录“已启用”布尔值；文件系统是真实来源。

3. 目标存在真实目录时绝不直接删除或覆盖；先备份、再建立 Junction，并保留回滚记录。

4. MVP 可继续以 Codex 中的真实 Skill 为源，OpenCode 等 Agent 建立 Junction；后续再升级为独立中央仓库。
